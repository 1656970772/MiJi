# AGENTS.md

## 默认语言与文档规则

- 默认使用中文与用户沟通和回答，除非用户明确要求使用其他语言。
- 默认使用中文生成文档标题、文件名和正文内容，除非用户明确要求使用其他语言，或项目现有规范明确要求使用其他语言。

## 图片显示规则

- 最终绘画、导出、截图、对比和工作产物默认仍保存为 `.png`；不要因为对话展示需要而把正式产物改成 `.webp`。
- 如果 Codex 需要在对话中列出或展示本地图片，额外从正式产物复制/转换一份 `.webp` 展示副本，再使用 `.webp` 文件路径显示到 Codex 对话中。
- 对话展示用 `.webp` 副本优先放到无空格短路径，例如 `C:\CodexPreview\preview.webp`，并在 Markdown 中写成 `![预览](C:/CodexPreview/preview.webp)`。
- 不要在对话嵌图路径里使用空格、尖括号、`%20` 等 URL 编码或相对路径；如果原始产物路径有空格，先复制/转换到 `C:\CodexPreview` 再展示。
- 对话最终回复里嵌入预览时优先引用 `.webp` 展示副本；同时保留并汇报对应 `.png` 正式产物路径。

## Git 与 GitHub 发布规则

- 遇到 Git 或 GitHub 相关请求时，先检查当前目录是否已经是 Git 仓库。
- 如果需要依赖 GitHub CLI，先执行 `gh auth status`，确认已登录且可用。
- 如果当前目录还不是 Git 仓库，先确认当前工作区就是要发布的目标内容，再执行初始化、首次提交与远程创建。
- 默认分支优先使用 `main`。
- 如果用户没有指定，默认是 `public` 仓库。
- 遇到 GitHub 发布、创建仓库、提交文件、更新文件等请求时，直接使用 GitHub API 完成远程操作，不要先尝试 `git push`、`git ls-remote` 或其他替代发布方式。
- 如果是发布单个文件、文档或少量内容，也直接使用 GitHub API 写入目标仓库和目标分支，不走其他发布路径。
- 不要把某一次任务的仓库名、提交信息、项目路径或远程地址写成全局默认值。
- 发布完成后，先验证远程地址、默认分支、远程提交状态，以及本地是否已建立正常跟踪关系，再向用户汇报结果。

## 本项目 GitHub 同步规则

- 当前项目每次执行完任务后，默认全量上传到 GitHub 仓库 `MiJi`。
- 仓库默认公开，默认分支使用 `main`。
- 上传应覆盖当前项目完整内容，包含 `Memory`、`raw` 和根目录文档；除非用户明确要求，不单独挑选少量文件上传。
- GitHub 远程操作默认继续遵守“使用 GitHub API，不先用 git push”的规则。

## 对话记录规则

- 默认在每个项目的根目录创建 `Memory` 文件夹，用于保存当前项目的对话记录。
- 对话记录文件默认使用创建时间命名，格式建议为 `yyyy-MM-dd_HH-mm-ss.md`。
- 每个对话记录文件最多保留 500 行内容。
- 单个对话不得拆分到两个不同的记录文件中；如果预计当前对话写入后会超过 500 行，应为下一次新对话创建新文件，而不是拆分当前对话。

## 来源与观点标注规则

- 所有事实性陈述、判断、建议和结论，默认都要提供来源支持，并在回答中明确标注来源。
- 来源优先使用用户提供材料、当前项目文件、官方文档或明确可核验的资料。
- 如果某一部分属于基于已有信息的分析、推断或方案设计，必须明确标注为“我的观点”或“我的判断”，并说明依据。
