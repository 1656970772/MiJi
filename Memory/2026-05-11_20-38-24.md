## 外部来源核验

补充核验来源：

- Epic 官方 Parallel Rendering Overview 文档说明，早期 renderer 在 Render Thread 上运行，Game Thread 会把命令入队，稍后在帧中执行；这些命令会调用 RHI 层。
- 同一官方文档说明，Render Thread 作为 frontend 入队平台无关图形命令，RHI Thread 作为 backend 通过合适的图形 API 转换/执行。
- 同一官方文档说明，UE 通常配置为 `single frame behind` renderer，即 Render Thread 处理 Frame N 时 Game Thread 处理 Frame N+1。
- Epic API 文档提供 `FRunnable` 与 `FRunnableThread` 页面，可作为原文线程执行体和线程封装概念的官方入口。

## 我的判断

这篇资料适合归档到 `Unreal Engine / Rendering / Multithreading / Source Reading`。

核心价值是把 UE 渲染源码中最容易混淆的线程边界讲清楚：游戏线程负责游戏状态，渲染线程组织渲染逻辑和平台无关命令，RHI 线程负责图形 API 后端转换。它尤其适合帮助读者理解 `ENQUEUE_RENDER_COMMAND` 为什么是跨线程边界，以及为什么渲染侧需要 Proxy 数据。

依据：知乎原文目录围绕多线程分配、任务系统、线程通信、资源竞争和同步展开；Epic Parallel Rendering 官方文档佐证了 Game Thread、Render Thread、RHI Thread、RHI 层和 single-frame-behind renderer 的基础关系。
